# 02 — Raspberry Pi 5: Network Intelligence Device

**Unbound · AdGuard Home · Prometheus · Grafana**

---

## Overview

This document covers the complete setup of the Raspberry Pi 5 as a dedicated network intelligence device. The Pi serves two primary purposes: recursive DNS resolution via Unbound, and network monitoring and observability via Prometheus and Grafana. This follows the single-purpose device philosophy - the Flint 2 handles network control, the Pi handles network observation.

The Pi replaced a previous Pi-hole setup that was abandoned due to a DNS loop issue (detailed in Issues Encountered). AdGuard Home on the Flint 2 took over blocking duties, with the Pi repurposed for recursive resolution and monitoring.

---

## Architecture

```
Device
  └── AdGuard Home (Flint 2, port 53)      ← blocking + client visibility
        └── Unbound (Pi 5, port 5335)       ← recursive resolution
              └── Root Servers              ← authoritative DNS
```

AdGuard Home handles ad, tracker, and malicious domain blocking at the network level. Unbound resolves queries recursively by walking the DNS tree from root servers - no upstream resolver ever sees the network's complete query history.

The monitoring stack runs alongside Unbound on the same Pi. Prometheus scrapes metrics from the node exporter (system health) and the Unbound exporter (DNS stats). Grafana visualizes both in real-time dashboards.

**Pi static IP:** `192.168.8.196`

---

## Tool Selection Rationale

### AdGuard Home vs. Pi-hole

Pi-hole was the original choice but was abandoned after a DNS loop caused 35 million queries overnight and nearly exhausted the Pi's RAM (see Issues Encountered). AdGuard Home was chosen as its replacement - it runs natively in GL.iNet's firmware, has a cleaner UI, supports per-client rules, and eliminates a network hop by running directly on the router.

### Unbound

Rather than forwarding to Cloudflare (`1.1.1.1`) or Google (`8.8.8.8`), Unbound resolves queries recursively from root servers. No third party sees the network's complete DNS history. Once the cache is warm, performance is comparable to forwarding resolvers. The Pi 5's 4-core CPU is put to use via Unbound's threading configuration.

### Why Not NextDNS

NextDNS was considered and rejected. It introduces a third-party dependency for query logs, requires trusting an external service with the network's complete DNS history, fails if NextDNS has an outage, and cannot resolve local hostnames. The goal is owning the stack end to end.

### Prometheus + Grafana

ntopng was the original monitoring candidate but was ruled out - its official repository has no release for Debian Trixie (the base for current Raspberry Pi OS Lite). Prometheus with node exporter and Unbound exporter was chosen instead. Grafana's dashboard ecosystem provided pre-built dashboards rather than requiring custom builds.

## Setup Process

### 1. OS Baseline

**Raspberry Pi OS Lite (64-bit, Trixie)** - headless, no desktop environment. All interaction via SSH.

Initial hardening:
- SSH key-based authentication (ed25519) - password auth disabled
- `fail2ban` - 5 retries / 10 minute window / 1 hour ban
- `unattended-upgrades` - automatic security patching
- `/etc/hosts` hostname fix to eliminate `sudo` warnings

### 2. Unbound

Installed from Debian repositories. Custom config at `/etc/unbound/unbound.conf.d/pi.conf`.

Key configuration decisions:
- Listens on port `5335` - port 53 is occupied by AdGuard Home on the router
- Listens on `0.0.0.0` to accept queries from the router
- Access control restricted to `127.0.0.1/32` and `192.168.8.0/24`
- 4 threads configured to match Pi 5 core count
- Extended statistics enabled for Prometheus metrics collection
- Remote control via Unix socket at `/run/unbound.ctl`
- DNSSEC hardening: `harden-glue`, `harden-dnssec-stripped`
- DNS rebinding protection via `private-address` blocks
- `prefetch: yes` to keep popular records warm in cache

Root hints file downloaded from `internic.net` to bootstrap recursive resolution from authoritative root servers.

### 3. AdGuard Home Configuration

AdGuard Home on the Flint 2 pointed at the Pi as its upstream DNS server (`192.168.8.196:5335`), completing the chain.

**Blocklists:**
- HaGeZi Pro - ads, trackers, telemetry
- HaGeZi Threat Intelligence Feeds - known malicious domains
- HaGeZi Windows Blocklist - Microsoft telemetry
- OISD Big - supplemental coverage

Per-client identification uses MAC address rather than IP, ensuring device identity persists across DHCP lease renewals. MAC randomization on mobile devices should be set to persistent for the home network to maintain consistent client tracking.

### 4. Prometheus

Installed from Debian repositories. Two exporters configured as scrape targets:

- `prometheus-node-exporter` (`localhost:9100`) - Pi system metrics: CPU, RAM, disk, network
- `unbound_exporter` (`localhost:9167`) - Unbound DNS metrics: query rates, cache stats, resolution times

The `unbound_exporter` binary was built from source (`github.com/letsencrypt/unbound_exporter`) — no pre-built arm64 package exists for the current version. Built using the Go toolchain in ~30 seconds. Runs as a systemd service with root privileges (required to access the Unbound Unix socket).

### 5. Grafana

Installed from the official Grafana apt repository. Two dashboards imported:

- **Dashboard 1860** - Node Exporter Full (Pi system health)
- **Dashboard 21006** - Unbound DNS metrics

Prometheus added as a data source at `localhost:9090`.

## Issues Encountered

### Pi-hole DNS Loop (Pre-existing)

**Problem:** Pi-hole was running via Docker. The router was querying Pi-hole as its DNS server, and Pi-hole was configured to use the router as its upstream - creating an infinite loop. This resulted in 35 million DNS queries overnight, nearly exhausting the Pi's RAM.

**Resolution:** Pi-hole and Docker were abandoned entirely. AdGuard Home on the Flint 2 replaced Pi-hole. The loop cannot occur in the new architecture since Unbound resolves from root servers rather than forwarding back to the router.

---

### Unbound Not Listening on Correct Port

**Problem:** After initial configuration, Unbound was listening on port 53 (default) rather than the configured port 5335. The config file was being read correctly but a service restart was required to apply the new port binding.

**Resolution:** `sudo systemctl restart unbound` applied the configuration correctly. A restart-required situation, not a config error.

---

### Unbound Refusing External Queries

**Problem:** Unbound was listening on `0.0.0.0:5335` but returning `REFUSED` / `recursion requested but not available` when queried from the Pi's network IP (`192.168.8.196`). AdGuard Home could not connect.

**Resolution:** Added `access-control` directives to the config: `127.0.0.1/32 allow` and `192.168.8.0/24 allow`. By default Unbound only accepts queries from localhost unless explicitly configured otherwise.

---

### ntopng Unavailable for Debian Trixie

**Problem:** The ntopng official repository has no release for Debian Trixie. Attempting to add the repository resulted in a 404 error.

**Resolution:** Pivoted to Prometheus + Grafana, available via the official Grafana apt repository and Debian repos respectively. This turned out to be the better long-term choice.

---

### unbound_exporter — No Pre-built arm64 Binary

**Problem:** The `letsencrypt/unbound_exporter` project does not publish a pre-built arm64 binary or `.deb` package for the current release (v0.5.0). The v0.4.6 `.deb` referenced in various guides no longer exists.

**Resolution:** Built from source using Go: `go install github.com/letsencrypt/unbound_exporter@latest`. The Go toolchain compiled the binary in approximately 30 seconds. Binary moved to `/usr/local/bin` and wrapped in a systemd service.

---

### Prometheus YAML Syntax Error

**Problem:** After adding the Unbound scrape job to `prometheus.yml`, Prometheus failed to start with `mapping values are not allowed in this context` on line 46. The `job_name` entry was missing a space after the colon (`job_name:'unbound'` instead of `job_name: 'unbound'`).

**Resolution:** Fixed the YAML syntax. YAML is whitespace and punctuation sensitive - a single missing space breaks the entire config.

---

### Grafana Dashboard DS_PROMETHEUS Variable Missing

**Problem:** Dashboard 21006 (Unbound) displayed `datasource DS_PROMETHEUS not found` for several panels after import. The dashboard references a template variable called `DS_PROMETHEUS` but does not create it automatically on import.

**Resolution:** Manually created the variable in Grafana: Dashboard Settings → Variables → New Variable. Type: Data source, Name: `DS_PROMETHEUS`, Data source type: Prometheus. Resolved all affected panels immediately.

---

### Loki/Promtail Package Corruption

**Problem:** Loki and Promtail were installed to test the `ar51an/unbound-dashboard` which requires both Prometheus and Loki. After deciding to use dashboard 21006 (Prometheus only), attempting to remove Loki and Promtail with `apt purge` failed - Promtail was in a very bad inconsistent state according to apt.

**Resolution:** Force removed with `sudo dpkg --remove --force-remove-reinstreq promtail` and `loki`. The `--force-remove-reinstreq` flag bypasses the consistency check. Packages were fully removed. This potentially could've caused issues, but luckily was avoided.

---

### Unbound Config Block Nesting Error

**Problem:** When adding the `remote-control` block and `extended-statistics` setting to `pi.conf`, the `remote-control` section was accidentally nested inside the `server:` block rather than defined as its own top-level section. This caused Unbound to fail to start.

**Resolution:** Corrected the config file structure - `remote-control:` must be a top-level block, not indented under `server:`. Verified with `sudo unbound-checkconf` before restarting.

## Configuration Reference

### Unbound (`/etc/unbound/unbound.conf.d/pi.conf`)

```yaml
server:
    verbosity: 0
    interface: 0.0.0.0
    port: 5335
    do-ip4: yes
    do-ip6: no
    do-udp: yes
    do-tcp: yes
    root-hints: "/var/lib/unbound/root.hints"
    harden-glue: yes
    harden-dnssec-stripped: yes
    use-caps-for-id: yes
    edns-buffer-size: 1232
    prefetch: yes
    num-threads: 4
    extended-statistics: yes
    private-address: 192.168.0.0/16
    private-address: 172.16.0.0/12
    private-address: 10.0.0.0/8
    access-control: 127.0.0.1/32 allow
    access-control: 192.168.8.0/24 allow

remote-control:
    control-enable: yes
    control-interface: /run/unbound.ctl
    control-use-cert: no
```

### Prometheus Scrape Config (`/etc/prometheus/prometheus.yml`)

```yaml
  - job_name: node
    static_configs:
      - targets: ['localhost:9100']

  - job_name: 'unbound'
    static_configs:
      - targets: ['localhost:9167']
```

### Unbound Exporter Service (`/etc/systemd/system/unbound_exporter.service`)

```ini
[Unit]
Description=Unbound Exporter
After=network.target unbound.service

[Service]
ExecStart=/usr/local/bin/unbound_exporter -unbound.host unix:///run/unbound.ctl
Restart=always
User=root

[Install]
WantedBy=multi-user.target
```
