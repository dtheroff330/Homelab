# 05 — Lenovo ThinkPad T400s: SSH Management Terminal

**Debian · XFCE · LUKS · SSH Key Auth · Device Purpose Specialization**

---

## Overview

The Lenovo ThinkPad T400s serves as the dedicated SSH management terminal for the homelab. Its sole purpose is to provide a reliable, hardened interface for managing network devices and servers - including the Raspberry Pi 5, mini PC, GL.iNet Flint 2 router, and AT&T BGW320-505 gateway.

The machine was deliberately chosen for device purpose specialization: one device, one job. It was also selected for its legendary ThinkPad keyboard - a critical factor for a machine that will see heavy terminal use - and for the opportunity to repurpose aging hardware rather than let it become e-waste.

---

## Hardware

| Component | Spec |
|---|---|
| CPU | Intel Core 2 Duo P9400 (2.4GHz, 64-bit) |
| RAM | 8GB DDR3 SO-DIMM (PC3-10600S, 204-pin) - upgraded from 4GB |
| Storage | 1.8" SATA (original to machine) |
| Display | 14.1" TN panel |
| Firmware | Legacy BIOS - no UEFI, no Secure Boot |

### Acquisition Notes

The machine arrived as a refurbished unit. Notable findings on arrival:

- Drive cover missing - cosmetic only, no functional impact
- Original drive contained Windows XP - confirmed hardware was functional before wiping
- Drive is a 1.8" SATA form factor, not standard 2.5" - limits upgrade options
- Battery dead on arrival - irrelevant, machine runs on AC power permanently

### RAM Upgrade

The original 4GB was insufficient for running a desktop environment alongside terminal sessions and a browser. The upgrade to 8GB (2×4GB, PC3-10600S SO-DIMM) was straightforward - a single panel on the bottom chassis, two screws, slots immediately accessible. The machine now boots to approximately 850MB RAM usage at idle under XFCE.

**Important:** the T400s requires SO-DIMM (204-pin laptop form factor), not UDIMM (240-pin desktop form factor). These are physically incompatible.

### Potential Future Upgrades

- **CPU:** Socketed PGA478 - compatible with P9700 (2.53GHz) for a modest performance bump
- **Thermal paste:** Nearly 20-year-old paste warrants replacement for better thermal headroom
- **WiFi card:** Stock card is dated; a newer Intel card would improve speeds and Linux compatibility
- **Storage:** 1.8" SATA SSDs exist but are expensive due to obscurity; an mSATA adapter is an alternative if a slot is available
- **BIOS:** This machine is capable of being librebooted and this is an idea I'm considering. This will not impact this machine's specific threat model, but it would be a challenging experiment - especially due to lack of documentation for the T400s.

## Software & Configuration

### Operating System

**Debian Stable with XFCE.** Chosen for its minimal footprint, stability, and alignment with the broader homelab philosophy of lightweight, Debian-based systems. MX Linux was evaluated but Debian Stable is a cleaner fit for a purpose-built machine - no extras, just the base.

- LUKS full-disk encryption enabled
- No persistent logins
- ~650MB RAM usage at first boot

### Browser

Firefox installed via the Mozilla repository (not the Debian ESR package). Chosen over Brave for better performance on the Core 2 Duo - Chromium-based browsers tend to be heavier on older single-threaded hardware.

Firefox use on this machine is intentionally restricted to local web UIs only:

- GL.iNet Flint 2 admin panel
- AdGuard Home dashboard
- Grafana monitoring dashboards
- Vaultwarden admin panel
- OpenMediaVault (NAS - planned)
- Nextcloud, Jellyfin, Immich (mini PC - planned)

Restricting browser use to local network addresses eliminates the primary real-world Spectre attack vector (malicious JavaScript from external sites) without sacrificing the utility of having a browser for managing the homelab stack.

### Security Hardening

- SSH key-only authentication (ed25519) - password auth disabled
- `fail2ban` - consistent with the rest of the homelab stack
- `unattended-upgrades` - automatic security patches
- `ufw` firewall - minimal ruleset, only necessary ports open
- LUKS full-disk encryption - protects against physical access threats
- No persistent logins - reduces exposure if the machine is left unattended

## Role in Homelab Architecture

The T400s holds SSH private keys for all headless devices in the homelab. Both the Pi 5 and the mini PC are hardened to accept key-only authentication - the T400s is the only machine capable of SSHing into them without physically connecting a monitor and keyboard. This creates a deliberate chokepoint: access to the homelab's core infrastructure flows through one trusted, hardened machine.

**Managed devices:**

| Device | Address | Role |
|---|---|---|
| Raspberry Pi 5 | `192.168.8.196` | Unbound DNS, Prometheus/Grafana |
| Mini PC | DHCP reserved | Docker services host |
| GL.iNet Flint 2 | `192.168.8.1` | Router, AdGuard Home, Tailscale |
| AT&T BGW320-505 | `192.168.1.254` | ISP gateway (IP passthrough mode) |

### Key Backup

SSH private keys are backed up encrypted to a durable USB drive stored offline. Loss of the T400s or its keys without a backup would require physical access to each headless device to restore SSH access. The backup is encrypted so that physical discovery of the USB drive does not translate into network access.

## Security Considerations & Threat Model

### Spectre & Meltdown

The Core 2 Duo P9400 is vulnerable to some Spectre variants. Intel did not release microcode updates for processors of this generation, and the kernel-level mitigations available for newer hardware do not fully apply here. This is an accepted risk given the actual threat model:

- Machine is on a local network with no inbound internet-facing connections
- Browser use is restricted to local web UIs - no external JavaScript execution
- No untrusted code runs on the machine
- Spectre attacks are primarily a concern in shared compute environments or against machines executing arbitrary external JavaScript

The combination of local-only browser use and no inbound external access makes the practical Spectre risk negligible for this deployment. Applying mitigations that don't match the actual attack surface is complexity without benefit.

### Physical Security

LUKS encryption protects against data exposure if the machine is physically compromised. No persistent logins reduce the window of opportunity if the machine is left unattended. The machine runs on AC power at a fixed location, reducing the likelihood of it leaving the environment.

## Lessons Learned

### Hardware

- Always verify SO-DIMM vs. UDIMM when purchasing laptop RAM - physically incompatible form factors
- 1.8" SATA is a niche form factor that significantly limits upgrade options compared to standard 2.5" SATA
- Legacy BIOS machines require MBR-compatible bootable USBs - UEFI-only flash tools produce drives that won't appear in the boot menu
- ThinkPad keyboards from the T400 era represent a meaningful quality difference over modern laptop keyboards for heavy terminal use

### Philosophy

- Hardware limitations enforce patience and intentionality - when every operation has a cost, you think more carefully about what you're doing
- Repurposing old hardware is both practically sound and philosophically aligned with FOSS and sustainability values
- Device purpose specialization produces cleaner, more auditable systems than consolidation
- A machine you enjoy using for its intended purpose reduces friction and improves workflow

### Security

- Threat model should drive security decisions - mitigations that don't match the actual attack surface are complexity without benefit
- Key-only SSH auth with a single trusted management machine is a clean, auditable access control model
- Restricting browser use to local UIs is an effective practical mitigation that doesn't require sacrificing browser utility entirely

## Summary

The T400s is a nearly 20-year-old ThinkPad running Debian Stable, hardened, encrypted, and dedicated to a single purpose: managing the homelab. The hardware upgrade (8GB RAM) and OS choice (minimal Debian + XFCE) make it comfortable for its workload, while the security configuration - key-only SSH, LUKS, fail2ban, local-only browsing - reflects a pragmatic approach to the real threat model rather than security theater.

This build also reflects the value of giving old hardware a purposeful second life. The T400s was not destined for a landfill - it was destined for a keyboard-heavy job that plays directly to its strengths.
